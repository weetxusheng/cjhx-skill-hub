# 工程实现手册

## 目的
本手册是 Skill Hub 的前后端实现、代码细节、UI 一致性、评审重点与质量门禁的统一入口。实现规则以这里为主，不再拆成多份平行文档。

## 通用工程原则
- 先看 `docs/10-architecture/` 与 `docs/30-product-flows/`，再按本手册落实现细节。
- 权限判断固定为：全局权限决定页面/接口可达，skill 级授权决定具体 skill 是否可操作。
- 后台写操作必须写 `audit_logs`，关键状态变化必须带缓存失效策略。
- 新增页面、抽屉、弹窗前，先补或同步补 `page-spec-template.md` 对应规格。
- 关键页面与关键区块必须具备 loading / empty / error / forbidden 四态；禁止空白页。
- 代码注释与 docstring 默认使用简体中文，只解释意图、边界、契约和副作用，不复述代码。

## 前端架构与页面实现
### 页面与目录落点
- 页面组件放 `apps/*/src/pages/**`，页面私有区块放同目录 `_components/`。
- 通用组件只承载复用视觉或交互，不耦合某个页面的路由、权限和状态机。
- 页面容器负责：路由参数、query state、mutation 编排、区块拼装。
- 区块组件负责：局部 JSX、局部状态、单一交互单元。
- 同一页面出现第二次重复逻辑时，优先抽共享 hook 或共享区块。

### 页面类型规则
- 列表页：必须有筛选区、结果区和空态/错误态，不允许裸表格。
- 详情页：先摘要与状态，再长内容与辅助数据。
- 工作台页：优先突出下一步动作，不把关键操作藏到深层详情。
- 抽屉/弹窗：关闭后必须保留原列表上下文。

### 权限与会话
- 菜单显隐、路由可达、按钮可用统一基于同一权限判断函数。
- capability 优先以后端下发为准，前端不重复推断状态机。
- 登录态判断以 `accessToken + /auth/me` 双重确认，不只看本地 token。
- 退出登录时清理所有后台 UI 持久化状态。

## 前端代码细节与交互一致性
### TypeScript / React
- 禁止 `any`、`as unknown as`、条件调用 hooks。
- 组件长度建议不超过 `250` 行；超出即拆分或登记例外。
- `queryKey` 必须包含 token、分页、筛选、排序等影响结果的全部维度。
- 写操作统一 `useMutation`，成功后显式 `invalidateQueries`。
- 禁止 render 期间发请求、权限不足直接 `return null`、接口失败后静默吞错。

### UI 与交互
- 单区域主操作最多一个 `primary` 按钮。
- 列表筛选区按钮顺序固定：`查询 -> 重置 -> 其他次要动作`。
- 时间字段统一 `YYYY-MM-DD HH:mm`，状态统一 `Tag`。
- 页面区块、抽屉头部、统计块、表格操作区遵守统一留白：模块间 `16px`，页面大区块 `24px`。
- 后台详情页模块高度、卡片边界、按钮层级保持统一，不允许单个模块漂移。
- 所有提交类交互必须提供 `loading + 成功反馈 + 失败反馈`，禁止“按钮 loading 结束后无任何结果提示”。
- 普通保存类动作（如主档保存、版本文案保存、角色配置保存）默认不做二次确认，但必须防重复提交，并在成功后给出显式成功反馈。
- 高风险动作必须二次确认：删除、停用/启用、授权移除、审核通过/拒绝、发布、归档、回滚，以及任何会改变线上状态、权限范围或可见性的动作。
- 二次确认弹层必须写清楚影响范围，至少说明“会改什么”“谁会受影响”“是否会归档/失效/丢失当前状态”。
- 高风险动作失败时不得只保留控制台或局部静默失败，至少要有可见错误反馈；若动作在弹窗/抽屉中触发，错误应保留在当前上下文。

### README 与详情排版
- README 推荐结构：概述、能力、安装、使用、注意事项、更新记录。
- 后台技能详情页统一顺序：状态摘要 -> 主档信息 -> 版本/时间线 -> 授权对象 -> 数据统计 -> 明细记录。
- 版本详情统一顺序：摘要 -> manifest -> 文案编辑 -> README 预览 -> 审核记录。

## 后端分层与领域边界
### Route
- 只负责参数解析、依赖注入、权限依赖、响应包装。
- 禁止在 route 内做复杂 SQL、事务编排、缓存删改。

### Service
- 负责状态机、事务、审计、缓存失效、写模型。
- 方法结构推荐：`load -> guard -> mutate -> audit -> commit -> post_commit`。
- 一个 service 方法只做一个业务动作，禁止万能入口 + `action switch`。

### Repository
- 只做数据读写与结果整形，不做业务判断与副作用。
- 多表查询必须显式选字段，禁止 `select *`。
- 列表查询默认分页，聚合字段必须可追溯。

### 领域落点
- `skills`：主档读写、公开列表、详情摘要。
- `versions`：版本内容、README、usage guide。
- `reviews` / `releases`：提审、审核、待发布、发布、回滚。
- `grants` / `permissions`：skill 级授权与 capability 判定。
- `stats` / `engagement`：点赞、收藏、下载、趋势与明细。

## 后端代码细节、事务与安全
- service / repository / route 应用文件建议不超过 `400` 行；测试文件建议不超过 `600` 行。
- 并发争用点（审核、发布、回滚、授权）必须加锁或做版本校验。
- 禁止在事务中执行慢外部 I/O。
- service 抛领域错误，route 或统一异常处理映射 HTTP 状态。
- 错误 message 必须可读，`401/403/404/409` 要稳定区分。
- 缓存统一走 helper，key 命名统一；失效时机在事务提交后。
- 输入必须经 schema 校验，禁止拼 SQL 字符串。
- 下载/导出接口必须做权限 + 范围双校验。

## 注释与 docstring 规则
- 后端 `api/routes/`、`services/`、`repositories/`、`core/`、`db/`、`scripts/` 的公开函数必须有 docstring。
- 上述后端强制目录中的 docstring 至少写：意图、关键边界（权限/状态/异常）、副作用（审计/缓存/计数）。
- 后端 `core/`、`db/`、`models/`、`schemas/`、`scripts/` 的 Python 模块必须有模块级说明，明确职责和边界；`models/`、`schemas/` 不要求为每个字段单独写注释，但要求关键状态、枚举、JSON 字段可追溯。
- 页面组件或应用壳层入口附近必须写“交互约定”注释，明确四态和关键上下文恢复规则。
- 页面壳层中承载登录、单点登录、上传入口、全局会话恢复等关键动作的函数，必须补一句意图注释；不要求对每个一行工具函数都写注释。
- 前端 `src/` 默认按目录分层写注释：`pages/` 与 `App.tsx` 写“交互约定”，`components/` 与页面 `_components/` 写“组件约定”，`lib/` 写“模块约定”，`store/` 写“状态约定”。
- `main.tsx`、`vite-env.d.ts` 这类启动/声明文件不强制写约定注释；纯样式常量、小型字面量映射可复用所属模块的顶层说明。
- 禁止 `TODO/TBD/WIP/FIXME` 占位注释直接入库。

### 哪些改动必须补注释
| 变更类型 | 必补注释 |
|---|---|
| 新增或重构路由处理函数 | route docstring，说明接口意图、权限依赖、错误语义 |
| 新增或重构 service 公共动作 | service docstring，说明状态机边界、审计、副作用 |
| 新增或重构 repository 公共查询 | repository docstring，说明筛选、排序、分页或聚合口径 |
| 新增或重构 `core/`、`db/`、`scripts/` 公开函数 | docstring，说明运行时边界、异常与副作用 |
| 新增或修改 `core/`、`db/`、`models/`、`schemas/`、`scripts/` 模块 | 模块级说明，说明职责、输入输出边界和禁止事项 |
| 新增页面 / 抽屉 / 弹窗 / 应用壳层主容器 | 页面“交互约定”注释，说明 loading / empty / error / forbidden |
| 新增业务组件或页面私有区块 | 组件“组件约定”注释，说明职责、输入输出和关键显隐/禁用规则 |
| 新增 `lib/` 模块或共享类型模块 | “模块约定”注释，说明对外契约、边界和禁止事项 |
| 新增 `store/` 状态模块 | “状态约定”注释，说明持久化、恢复和清理规则 |
| 页面壳层中的登录、SSO、上传入口、全局会话恢复逻辑 | 在关键函数或 effect 前补一句意图注释，说明为什么存在 |
| 修改权限、状态机、统计逻辑 | 在对应函数或区块附近补“为什么这样判定”的边界说明 |
| 修改缓存失效、计数写入、审计写入 | 补副作用说明，不允许只看代码猜时机 |

### 可以不补或少补的场景
- 纯样式改动且不改变交互语义
- 文案调整且不改变状态、权限、流程含义
- 明显自解释的私有小工具函数
- 单纯的 JSX 结构抽取，但原有页面交互约定仍成立
- `models/`、`schemas/` 中字段级显式类型和 `Field(...)` 已足够表达含义时，可不再为每个字段重复注释

### 推荐模板
#### Python docstring
```python
def publish_version(...):
    """发布一个已审核通过的版本。

    意图：
      - 将目标版本切换为当前线上版本。
    边界：
      - 权限：要求具备 skill.publish。
      - 状态：仅允许 approved -> published。
    副作用：
      - 写 audit_logs
      - 失效 public skill detail 缓存
    """
```

#### Python 模块说明
```python
"""权限判定与 skill 级授权聚合。

- 职责：提供统一 capability 计算，不直接持有 HTTP 依赖。
- 边界：只读数据库，不写审计、不改缓存。
- 禁止：不要在这里编排事务或返回 HTTP 响应对象。
"""
```

#### 前端页面交互约定
```ts
/**
 * 交互约定：
 * - loading：展示 skeleton，保留页面骨架
 * - empty：展示 Empty 与下一步引导
 * - error：展示可重试错误提示
 * - forbidden：展示权限不足说明，禁止空白页
 */
```

### 反例
- `"""更新版本"""` 这种只有动作名、没有边界和副作用的 docstring，不算合格。
- 页面里只写“TODO: 补空态”，不算交互约定。
- 在多个文件复制同一段权限说明，而不指向唯一事实来源，不算合格注释。
- 用注释掩盖坏结构，例如“这里先这样写，后面再拆”，不算合格注释。

## 代码评审与质量门禁
### 评审优先级
1. 状态机错误
2. 权限越权或漏判
3. 统计口径错误
4. 文档与实现不一致
5. 测试缺失
6. 代码细节退化

### Review blocker
- 审核发布状态机可被绕过
- 权限缺失或错误
- 冗余统计与明细不一致
- 关键页面无错误态或权限不足态
- 命中“必须补注释”的改动却未补 docstring 或交互约定
- 长文件、重复实现或跨层职责混乱却未拆分/未登记
- 命中 lint 或质量门禁的问题未修复

### 文件拆分与共享边界
- 前端页面 > `250` 行、通用组件 > `180` 行建议拆分。
- 后端应用文件 > `400` 行建议拆分；测试文件 > `600` 行建议拆分。
- 满足以下任一条必须拆：多种页面逻辑混杂、3 组以上 query/mutation/form 状态、多个业务域挤在同一文件、长条件渲染链、第二次复制粘贴。
- 存量例外统一登记到 `code-quality-debt-register.md`，不得默许。

### 门禁命令
- `pnpm lint`
- `bash infra/scripts/check-code-quality.sh`
- `bash infra/scripts/check-comment-placeholders.sh`
- `bash infra/scripts/check-comment-contracts.sh`
- `infra/scripts/release-check.sh`

## 完成定义
- 实现符合分层与页面结构规则。
- 注释、docstring、页面规格和文档同步完成。
- lint、质量门禁和对应测试层级通过。
