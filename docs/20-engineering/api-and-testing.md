# API 与测试规范

## API 设计
### 统一响应
- `code`
- `message`
- `data`

### 输入输出要求
- 列表接口优先返回结构化字段，不把前端需要解析的文本塞进单列
- 分页接口统一包含：
  - `items`
  - `total`
  - `page`
  - `page_size`
- 工作台类接口必须返回足够的摘要信息，避免前端 N+1 请求
- 详情接口若存在“按钮是否可用”“明细是否可见”等判断，应优先由后端返回 capability 字段，避免前端重复推断权限和状态

### 错误响应
- 权限错误返回可读 message
- 状态冲突返回清晰 detail，不能只给通用失败文案

## 测试层级
详细执行流程、命令分层、手工 smoke 和 Agent 输出格式统一以 `docs/20-engineering/feature-test-playbook.md` 为准；本文件只保留测试总纲和发布阻断原则。

### 后端
- 单接口回归
- 状态机链路测试
- 权限矩阵测试
- 测试过程中动态创建数据库场景数据
- 统计口径测试

### 前端
- 构建通过
- 核心页面 smoke
- Playwright 覆盖：
  - 审核中心
  - 待发布
  - skill 详情
  - portal 详情抽屉

## 功能完成后的默认执行规则
- 任何功能开发、修复、重构完成后，都必须先按 `docs/20-engineering/feature-test-playbook.md` 判断变更类型
- 开发期间默认保持项目在线，普通改代码依赖 `uvicorn --reload` 和 Vite 热更新，不把整套停服重启作为默认动作
- 在进入手工 smoke、联调或最终测试前，必须先执行 `pnpm local:ensure`，确认本地项目仍在线
- 再根据命中的变更类型选择 `L1`、`L2`、`L3` 测试层级
- 命中权限、状态机、数据库、API、统计、发布相关改动时，必须自动升级到更高测试层级
- 若有测试未执行，必须在结果中说明阻塞原因与残余风险

## 当前必须覆盖的回归线
- 上传 -> 提审 -> 审核 -> 待发布 -> 发布 -> 回滚
- skill 级授权限制
- owner / maintainer / reviewer / publisher 的 capability 与按钮对齐
- 收藏 / 点赞 / 下载统计口径
- 下载明细脱敏
- 处理记录展示
- 待发布队列展示
- 用户、角色、审计日志分页与筛选

## 发布阻断项
以下任一失败都视为阻断发布：
- 后端全量 pytest 失败
- 前端构建失败
- 关键 Playwright smoke 失败
- migration 无法从空库升级
- 待审核 / 待发布 / 处理记录 页面打不开

## Smoke Checklist
- 管理员可登录后台
- 技能列表正常打开
- 审核中心能看到 `submitted`
- 待发布能看到 `approved`
- 处理记录能看到最近动作
- portal 能打开技能详情并下载

## 测试结果要求
- 最终测试结论只能是：`可合并`、`仅本地通过`、`不可交付`
- 结论必须附带已执行测试摘要
- 不能静默跳过未执行项
