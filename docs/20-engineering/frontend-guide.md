# 前端开发规范

## 通用要求
- 页面必须补：
  - loading
  - empty
  - error
  - forbidden
- 列表类页面必须支持筛选条件恢复
- 按钮显隐统一走权限判断，不允许回退到角色名硬编码
- 页面文案要体现业务状态，不要直接把数据库状态码原样暴露给用户

## 页面类型规则
### 列表页
- 必须提供必要筛选项
- 工作台列表优先展示“下一步动作所需信息”
- 行内动作应控制在 1-3 个，复杂操作进入详情或弹窗

### 详情页
- 顶部先展示摘要与状态，再展示长内容
- 必须让用户一眼区分：
  - 当前线上版本
  - 最新版本
  - 当前可执行动作

### 表单页 / 编辑区
- 权限不足时显示只读说明，不直接隐藏整个区块
- 保存失败时必须显示后端错误
- 对状态受限的字段要明确写“当前状态不可编辑”

### 工作台页
- 审核中心、待发布、处理记录属于工作台页
- 工作台页优先提供：
  - 筛选
  - 摘要信息
  - 高价值动作入口
- 不要求一次展示全部明细，但必须能进入详情页

## `portal-web`
- 用户第一视角优先看技能，不要把平台说明放在首屏之前
- 详情优先在当前上下文内打开，不打断探索
- 点赞、收藏、下载语义必须分离
- 上传入口存在，但不替代后台完整工作台

## `admin-web`
- 列表页优先承载工作台入口，不要把关键操作埋在版本详情深处
- 审核、待发布、处理记录必须是清晰的独立入口
- Skill 详情必须同时展示：
  - 主档信息
  - 版本与审核摘要
  - 授权对象
  - 数据统计
- 用户管理页应支持查看某个用户命中的 skill 授权，并标明直接授权与角色继承

## 权限驱动显示
- 菜单显示由全局权限决定
- Skill 详情中的操作按钮由“全局权限 + skill 授权”共同决定
- 若后端已经返回 capability 字段，前端必须直接消费 capability，不再本地重复推断状态机和 scope
- 没权限时优先显示只读和原因，不要让页面无声消失

## 组件复用边界
- 共用交互语义必须抽成组件，例如：
  - 版本动作弹窗
  - 上传弹窗
  - 统计卡片
- 不要把仅一页生效的特殊视觉逻辑写进全局样式

## UI 一致性要求
- 状态显示统一使用 `Tag`
- 危险操作统一二次确认或带说明弹窗
- 表格字段命名和后端 schema 含义保持一致
- 同类按钮保持统一层级和颜色，不允许同一页面主按钮风格混乱
- 为方便定位与排障，页面内所有“区域块”（例如：`Card`、筛选区、列表 `Table` 容器、表单区块、`Modal/Drawer` 内容区等）都必须在区域根节点添加稳定的 `id` 属性；同一页面内保持唯一。`id` 使用 kebab-case，并以页面 slug 作为前缀，例如：`admin-users-filters`、`admin-users-table`、`admin-users-role-modal`。
