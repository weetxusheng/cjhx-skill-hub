# 前端专项手册

## 目的
本手册是 Skill Hub 前端实现的专项补充，覆盖页面结构、组件边界、样式系统、命名规则、可访问性、性能与测试基线。跨栈通用规则仍以 `engineering-handbook.md` 为主；凡是涉及前端目录落点、页面实现、样式规范、设计 token、交互一致性和页面测试的问题，以本手册为准。

## 适用范围
- `apps/portal-web/src/**`
- `apps/admin-web/src/**`

## 必读顺序
### 做前端页面、抽屉、弹窗
1. `engineering-handbook.md`

3. 对应页面规格

## 设计原则
- 前端首先服务业务闭环，不追求局部炫技。
- portal 与 admin 可以拥有不同视觉气质，但必须共享同一套可维护的实现原则。
- 页面必须让用户一眼理解“现在能看什么、下一步能做什么、失败了怎么办”。
- 视觉规则优先落成 token 和组件约束，不允许长期依赖局部魔法值。
- 能复用语义时先复用，不因追求复用而抽出无业务意义的“空组件”。

## 前端分层与目录边界
### 顶层目录职责
- `src/pages/`：路由级页面容器；负责路由参数、query state、mutation 编排、区块拼装。
- `src/pages/**/_components/`：仅服务当前页面的区块级组件；可承载复杂 JSX 和局部交互。
- `src/components/`：跨页面复用的业务组件或通用组件；不得隐式耦合某个页面路由。
- `src/lib/`：API client、数据转换、权限辅助、样式常量、展示映射等纯模块能力。
- `src/store/`：跨页面共享且需要持久化或集中管理的状态；不得塞业务页面私有状态。
- `src/assets/`：静态资源；禁止堆放可由代码生成的映射配置。

### 页面、组件、Hook 的边界
- 页面容器：
  - 允许：读路由、组装 query 参数、拼接 capability、组织页面区块。
  - 禁止：长列表列定义、复杂样式逻辑、重复的接口适配。
- 页面区块组件：
  - 允许：封装当前页面的一段完整视觉和交互。
  - 禁止：偷读全局路由状态、跨页面复用时仍藏在 `_components/`。
- 通用组件：
  - 允许：沉淀稳定交互语义，例如上传弹窗、版本动作弹窗、统计卡片。
  - 禁止：把页面特有条件和文案写死在组件内部。
- 自定义 Hook：
  - 允许：封装 query state、复杂 mutation 编排、可复用的列表/表单交互。
  - 禁止：既做数据请求又直接输出大段 JSX。

### 何时新增模块，何时扩已有模块
- 新页面出现独立业务语义时，优先新增页面目录。
- 仅为当前页面拆大文件时，优先放入对应页面 `_components/`。
- 复用次数预期大于等于 2，且交互语义稳定，才提升到 `src/components/`。
- API 响应转换、状态标签映射、颜色/图标映射类逻辑优先进 `src/lib/`，不要散落在页面 JSX 中。

## 命名规范
### 文件与目录
- 页面文件使用 `PascalCase.tsx`，名称表达业务语义，例如 `MarketplacePage.tsx`。
- 页面私有目录固定用 `_components/`，不要出现 `_common/`、`partials/` 这类模糊命名。
- Hook 文件使用 `camelCase.ts` 或 `useXxx.ts`，导出主 hook 与文件名一致。
- `lib/` 模块名优先使用业务语义，例如 `categoryVisuals.ts`、`portalApi.ts`。

### React 组件与 Hook
- 组件名使用 `PascalCase`，必须体现业务角色，禁止 `PanelA`、`Box2`。
- Hook 使用 `useXxx` 命名；若封装 query state，名称中应体现资源和场景，例如 `useSkillListQueryState`。
- 布尔变量以 `is` / `has` / `can` 开头，表达能力或状态，而不是视觉结果。

### API、查询键与状态
- Query key 统一以数组形式声明，第一段固定资源名，后续段按“身份 -> 条件 -> 排序/分页”排列。
- 能力字段统一使用 `capabilities` 或 `canXxx` 语义，禁止混用 `allow`、`enabled`、`permission`。
- URL query 参数与后端接口字段保持一致，不另起前端别名。

### CSS 类名
- 全局类名使用 kebab-case。
- 页面根类必须带页面语义前缀，例如 `marketplace-page`、`admin-skill-detail-page`。
- 区块类名以页面或组件语义起头，例如 `marketplace-toolbar`、`upload-center-record-list`。
- 状态类统一使用修饰后缀：
  - `is-active`
  - `is-selected`
  - `is-danger`
  - `is-empty`
- 禁止使用纯视觉名词做业务类名，例如 `red-box`、`left-card`、`big-title`。

### 设计 token 命名
- token 一律使用语义命名，不使用实现命名。
- 颜色 token 推荐结构：
  - `--color-bg-page`
  - `--color-bg-card`
  - `--color-text-primary`
  - `--color-text-secondary`
  - `--color-border-default`
  - `--color-accent-primary`
- 阴影 token 推荐结构：
  - `--shadow-xs`
  - `--shadow-sm`
  - `--shadow-md`
  - `--shadow-lg`
- 圆角 token 推荐结构：
  - `--radius-sm`
  - `--radius-md`
  - `--radius-lg`
  - `--radius-pill`
- 间距 token 推荐结构：
  - `--space-2`
  - `--space-4`
  - `--space-8`
  - `--space-12`
  - `--space-16`
  - `--space-24`
  - `--space-32`

## 样式系统与设计 token
### 总原则
- 视觉基础值优先沉淀为 CSS variables，不允许同类页面长期散落不同常量。
- 新增品牌色、文字色、阴影、圆角、边框前，先判断能否复用现有 token。
- 新值若被 2 个以上组件复用，必须提升为 token。
- 同一产品端内，组件不允许同时混用多套圆角和阴影体系。

### 颜色规范
- 色彩分层必须明确区分：
  - 页面背景
  - 卡片/容器背景
  - 边框
  - 主文本
  - 次文本
  - 品牌主色
  - 成功/警告/错误/信息状态色
- Portal：
  - 允许更强品牌感和氛围背景，但正文阅读区域必须保持高对比度。
  - 背景渐变只能服务氛围，不得影响卡片边界识别和正文可读性。
- Admin：
  - 优先稳定、克制、可读；避免大面积高饱和背景。
  - 工作台区域主色只用于主按钮、关键状态和聚焦控件。
- 禁止项：
  - 用纯颜色名直接表达语义，如“红色按钮=危险、橙色按钮=主要”但没有 token 说明。
  - 同页同时出现 2 套主按钮色。
  - 文本颜色依赖透明度堆叠去凑层级。

### 字号与排版规范
- 必须形成明确层级：
  - 页面标题
  - 区块标题
  - 卡片标题
  - 正文
  - 辅助文案
  - 数据标签/说明
- 推荐基础级别：
  - 页面标题：`36px` 左右
  - 区块标题：`24px` 左右
  - 卡片标题：`18px` 到 `20px`
  - 正文：`14px` 到 `16px`
  - 辅助文案：`12px` 到 `14px`
- 行高遵循：
  - 标题偏紧
  - 正文偏松
  - 标签与状态文本紧凑但不可挤压
- 同一区块内不得同时出现 3 种以上正文字号。

### 间距规范
- 仅允许使用固定 spacing scale，优先 4 的倍数体系。
- 推荐基线：
  - 区块内小间距：`8 / 12`
  - 表单项与卡片内容：`16`
  - 模块之间：`24 / 32`
  - 页面主区块间：`40 / 48`
- 同一详情页的主模块间距必须统一，不允许某块 `12px`、某块 `20px`、某块 `28px` 漂移。

### 圆角规范
- 小型控件：`--radius-sm`
- 输入框、按钮、普通卡片：`--radius-md`
- 大卡片、抽屉内大模块、品牌化面板：`--radius-lg`
- 胶囊、Tag、计数 pill：`--radius-pill`
- 同一个组件不要同时混用 2 种相近圆角。

### 阴影规范
- 阴影只承担层级提示，不承担全部边界表达。
- 卡片边界必须是“边框 + 阴影 + 背景”协同，而不是只靠阴影。
- 推荐层级：
  - `--shadow-xs`：轻微悬浮，如按钮 hover
  - `--shadow-sm`：普通卡片
  - `--shadow-md`：浮层、抽屉中的重点模块
  - `--shadow-lg`：极少数品牌区域，不用于普通表单
- Portal 可适度更柔和，Admin 以克制清晰为主。

### 边框与分隔
- 卡片、筛选栏、输入框、抽屉内模块必须有明确边界，不能与背景糊成一片。
- 分隔线颜色必须弱于主边框，不能抢正文。
- 需要强调 hover/active 时，优先提升边框和阴影，不优先改大面积背景色。

### 何时允许原始值
- 只允许在以下场景使用原始值：
  - 第一次引入 token 时定义 token 本身
  - 第三方组件覆盖必须用到的极少量兼容值
  - 单次实验性样式，且当轮会立刻沉淀进 token
- 其余情况禁止在业务样式里直接散落十六进制色值、随机圆角和随机阴影。

## 页面与交互规范
### 列表页
- 结构固定为：筛选 -> 摘要 -> 列表/卡片 -> 分页/动作。
- 筛选工具栏内部控件必须统一高度、对齐方式和边框语义。
- 卡片列表必须让“标题、核心状态、下一步动作”一眼可见。
- 筛选条件切换后，分页和路由参数必须同步恢复。

### 详情抽屉与详情区
- 顶部先给摘要与主动作，再展示长内容。
- 抽屉内部模块必须保持同一宽度节奏、标题层级和卡片容器语义。
- 抽屉/详情不能出现纯白区域与背景糊在一起的“无边界模块”。

### 弹窗与表单
- 弹窗只承载当前上下文动作，不替代完整工作台。
- 表单字段顺序优先业务顺序，不按后端字段顺序机械平铺。
- 保存、提交、发布等动作必须提供：
  - loading
  - 防重复提交
  - 成功反馈
  - 失败反馈
- 删除、禁用、授权移除、审核通过/拒绝、发布、归档、回滚必须二次确认。

### 状态矩阵
- 每个关键页面、抽屉、弹窗都必须补齐：
  - loading
  - empty
  - error
  - forbidden
- forbidden 必须解释原因和建议动作，不能直接消失。
- empty 要区分“真没有数据”和“筛选后无结果”。

## 可访问性规范
- 所有可点击元素必须可通过键盘访问。
- 抽屉、弹窗打开后焦点应进入容器内部，关闭后返回触发按钮附近。
- 只用颜色表达状态时，必须同时提供文字或图标。
- 重要操作按钮的 hover、focus、disabled 状态必须可见。
- 表单错误必须贴近字段呈现，不允许只在顶部弹一句模糊提示。
- 图标按钮若无可见文本，必须提供 `aria-label`。

## 性能与可维护性规范
- 出现 Vite chunk warning 的页面或模块，必须评估是否拆出懒加载区块。
- 列表页数据量较大时优先后端分页；前端不得一次渲染无上限列表。
- Markdown、长详情、复杂统计区优先拆分成惰性区块，不把全部内容塞进一个组件。
- 不要在 render 期间做大对象转换或复杂排序；应提前在 `lib/` 或 hook 中处理。
- 单文件过大时优先拆：
  - 页面容器 vs 区块组件
  - 表格列定义
  - 常量映射
  - 表单 schema / options

## 测试与稳定 id 规范
- 关键区域块必须有稳定 `id`，格式：`{page-slug}-{area-name}`。
- E2E 优先依赖稳定 `id`，避免依赖中文文案或临时类名。
- 前端改动至少考虑这几类验证：
  - 页面是否保留筛选和返回上下文
  - loading / empty / error / forbidden 是否齐全
  - 提交后是否有反馈与防重复提交
  - 高风险动作是否二次确认
  - 能力不足时按钮、菜单、页面是否一致

## 注释与说明要求
- `src/pages/**/*.tsx` 与 `src/App.tsx` 必须有“交互约定”注释。
- `src/components/**` 与页面 `_components/` 必须有“组件约定”注释。
- `src/lib/**` 必须有“模块约定”注释。
- `src/store/**` 必须有“状态约定”注释。
- 样式复杂、状态多、容易误用的组件，必须额外写明边界和禁止事项。

## 评审 blocker
- 同页出现多套主按钮样式、圆角体系或阴影体系。
- 大量直接写死十六进制色值、字号、阴影而没有 token 归属。
- 页面没有完整状态矩阵。
- 权限不足时页面或抽屉空白。
- 关键提交动作无 loading、无成功/失败反馈或可重复点击。
- 高风险动作无二次确认。
- 筛选栏、卡片、详情模块边界混乱，影响识别与操作。
- 组件、Hook、样式命名无法表达业务语义。

## 变更触发
以下变更必须同步回看本手册：
- 改 Portal 或 Admin 的主题、样式基调、token 命名
- 改页面结构、筛选工具栏、卡片体系、详情抽屉
- 改前端目录结构、组件分层、Hook 分层
- 改可访问性、焦点管理、E2E 选择器约定
- 改前端性能策略、拆包策略、长列表策略
